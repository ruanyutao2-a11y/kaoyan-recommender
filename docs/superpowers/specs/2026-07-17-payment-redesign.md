# 付费解锁改造设计

**日期**: 2026-07-17
**状态**: 待实现

## 问题

当前系统在付款前就生成并展示兑换码，用户可以不付款直接使用兑换码解锁。本质问题：系统无法验证用户是否真的在淘宝付了款。

## 方案

采用"首次免费 + 付费审核"混合模式：

- **首次评估**：免费解锁完整结果（24小时内有效），消除付费摩擦，最大化转化
- **24小时后回看 / 重复使用**：需付费 9.9 元
- **首次付费**：信任优先，自动立即解锁
- **同一设备多次付费**：需管理员手动审核

## 新付费流程

```
用户提交评估
     │
     ▼
  首次使用且在24h内？
     │
     ├─ YES → 直接看完整结果（标记 free_until = 创建时间+24h）
     │
     └─ NO → 展示付费墙
               │
               ├─ 展示微信/支付宝个人收款码（静态图片）
               ├─ 用户扫码付款 9.9 元
               ├─ 用户输入转账单号后4位
               └─ 提交
                    │
                    ├─ 该设备首次付费 → 自动解锁（信任）
                    └─ 该设备非首次付费 → 待管理员审核
```

## 技术变更

### 数据库变更

evaluations 表新增字段：
- `free_until TEXT` — 免费期截止时间，已付费为 NULL
- `unlock_type TEXT` — 'free' | 'first_pay' | 'paid' | 'manual_review'

orders 表新增字段：
- `txn_ref TEXT` — 用户填写的转账单号后4位
- `reviewed_by TEXT` — 审核人
- `reviewed_at TEXT` — 审核时间
- `notes TEXT` — 备注

### API 变更

| API | 方法 | 变更说明 |
|-----|------|---------|
| `GET /api/result/:id` | GET | 免费期内返回完整结果；过期返回预览 |
| `POST /api/order` | POST | 不再生成兑换码，改为记录付款申请 + 返回收款码 URL |
| `POST /api/redeem` | POST | **删除** |
| `POST /api/submit-payment` | **新增** | 用户提交付款信息（转账单号后4位） |
| `GET /api/admin/orders` | **新增** | 管理员查看待审核订单（需管理员密码 header） |
| `POST /api/admin/approve` | **新增** | 管理员审核通过/拒绝（需管理员密码 header） |

### 前端变更

| 组件 | 变更说明 |
|------|---------|
| `Paywall.tsx` | 重写：展示收款码 + 转账单号输入 + 自动/审核状态 |
| `PreviewPage.tsx` | 判断免费期，免费则直接放行完整结果 |
| `RedeemPage.tsx` | 删除 |
| `AdminPage.tsx` | **新增**，极简订单审核页（密码保护） |

### 管理后台

- 地址：`/admin`
- 密码保护，密码存 Worker 环境变量 `ADMIN_PASSWORD`
- 功能：查看待审核/已通过/已拒绝订单，一键通过/拒绝
- 安全：密码存 sessionStorage，关闭浏览器失效；API 通过 header 传密码

### 收款码

- 微信和支付宝个人收款码图片放入 `public/` 目录
- Worker vars 可配置收款码 URL，方便更新
